# Copilot Integration Guide

## What copilot-instructions.md Is

A file committed to the project repo at .github/copilot-instructions.md.
GitHub Copilot in VS Code and JetBrains Rider reads this file automatically for every suggestion.
It primes Copilot with your architecture, standards, and patterns so suggestions are consistent
with decisions made in the ICM workspace rather than generic .NET defaults.

## Where It Lives in the Project Repo

.github/
  copilot-instructions.md    <- Copilot reads this automatically in VS Code and Rider
docs/
  coding-standards.md        <- full standards doc, copied from Stage 03 output
  architecture.md            <- full architecture doc, copied from Stage 02 output
  checklists.md              <- operational checklists, copied from Stage 05 output

## VS Code Setup

1. Install GitHub Copilot and GitHub Copilot Chat extensions.
2. Commit .github/copilot-instructions.md to the project repo.
3. Copilot loads it automatically - no configuration needed.
4. In Copilot Chat, use: Ask Copilot to explain what context it has to verify it loaded.

## JetBrains Rider Setup

1. Install the GitHub Copilot plugin from JetBrains Marketplace.
2. Sign in with your GitHub account.
3. Commit .github/copilot-instructions.md to the project repo.
4. Rider picks it up via the Copilot plugin automatically from .github/ directory.
5. Use Copilot Chat in Rider the same as VS Code.

## Keeping It in Sync

- copilot-instructions.md is generated by Stage 05 and should be regenerated when:
  - Architecture decisions change (new ADR in Stage 02)
  - Coding standards are updated (Stage 03 re-run)
  - New services are added (Stage 04 re-run)
- Treat it like a compiled artifact: source of truth is the ICM workspace, not the file itself.
- Commit it to the project repo after each ICM pipeline run that changes standards or architecture.

## What Copilot Can Do With Full Context

- Suggest method signatures that follow your async and nullability patterns.
- Generate Blazor components using your preferred Telerik components and state management approach.
- Write error handling that matches your exception strategy.
- Suggest OTEL instrumentation that follows your observability rules.
- Warn inline when you ask it to do something that violates your stated anti-patterns.
- Generate unit tests that match your testing strategy from Stage 03.
